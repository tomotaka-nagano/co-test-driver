# 1. テストを実施するまでの進め方
0. もしかして要求仕様書とかユースケース記述とかから一様 (とまでは言わないが、なるべく一様) な入力を作るフェーズが必要？
   - ただこれってほぼ 1 と同義になそうな気もするから一旦ナシ
   - いや、フェーズを持つ必要はないが、内部的に整理・保持する仕組みは必要。更にはデバッグ用に見える仕組みも必要

1. 要求仕様書とかからテストシナリオを作る
   - 2.1 の資料 + シナリオ導出観点が必要
2. シナリオとテスト環境からテストケースと観点を作る
   - Q. 1.3 との違いは？ → 1.2 の方が人間が理解しやすい
   - 1.1 の結果 + 2.2 の資料
3. テストケースからテスト環境で実行可能なテストデータを作る
   - 1.2 の結果 + 2.2 の資料
4. テスト環境でテストを実行する
   - 1.3 の結果 + テスト環境
5. テスト環境から出力された結果と観点を突き合わせて正誤判定を行う
   - 1.4 の結果 + 1.2 の結果

# 2. 静的？に入力する情報
1. テストシナリオを考える源泉となる情報
   1. ユースケース記述
   2. 要求仕様書
2. テスト環境を使うための情報
   1. シミュレーター API など

# 3. 必要なツール (MCP的な意味で)
1. テストシナリオ導出
   1. イニシャル導出 (導出方法 + 確認観点)
   2. ユーザーの指摘を含む導出
      (導出方法 + 過去導出結果 + 確認観点)
      ※ 確認観点は上記と共通？でのユーザーに上書きされた場合も考慮が必要
